# Design

## Visual Identity

Dispatch is a calm, premium developer-product interface with two coordinated modes: an emotional editorial landing experience and a compact operational application UI. The brand should feel like infrastructure for serious human + agent software teams, but it should not look like a noisy sci-fi control room. The design language is restrained, sharp, and quietly confident.

The landing page leads with an almost cinematic first impression: a near-black shell, white serif headline, simple centered actions, and a large clean product screenshot. The screenshot itself is intentionally closer to a polished issue tracker than a fantasy dashboard. Dispatch should feel familiar to people who use Linear, GitHub Issues, Slack, and modern code editors, while still carrying a distinct agent-orchestration identity through the eight-point dispatch mark and the “humans + agents” product visuals.

## Brand Principles

1. **Editorial first, dashboard second.** The landing page should not start by overwhelming the user with product chrome. The headline, copy, and a single product preview carry the story.
2. **Operational clarity.** Inside the product, density is acceptable, but hierarchy must remain obvious: issue title, status, assignee, comments, and agent activity should be easy to scan.
3. **Restrained futurism.** Use subtle glow, atmospheric imagery, and agent status indicators sparingly. Avoid heavy neon, 3D panels, excessive glassmorphism, or command-center clichés.
4. **Humans and agents share the same surface.** Avatars, assignees, activity rows, and comments should make agents feel like teammates without making the UI playful or cartoonish.
5. **Open-source credibility.** Layouts should be practical, readable, and implementation-friendly. Decorative ideas must never compromise legibility.

## Color System

The core application palette is neutral and semantic: white or near-black surfaces, muted zinc-gray borders, and a blue brand accent. The landing experience adds atmosphere through deep ink backgrounds and cool scenic blues, with warm coral highlights used only as environmental color or minor visual warmth.

Use **#05070b** as the canonical landing darkness. It is darker and more cinematic than the application dark background. Use **#0a0d12** for light-section text and black buttons because it is slightly softer than pure black while still reading as premium.

The application should prefer semantic roles over saturated decoration:

- **Blue / info** for active agents, connected runtimes, and system information.
- **Green** for successful completion and healthy online states.
- **Yellow / orange** for warnings, priority, or pending review.
- **Red** for destructive and offline/error states.

Transparent whites define the landing hierarchy: white at high opacity for primary text, 50–60% opacity for supporting labels, and 10–18% opacity for borders. This keeps the dark hero elegant rather than boxed-in.

## Typography

Dispatch uses **Inter** for product UI and **Source Serif 4** for editorial landing headlines. This contrast is central to the visual identity. The serif headline gives the landing page confidence and memorability, while Inter keeps the operational product crisp and familiar.

Hero headlines should be very large, tightly tracked, and line-height compressed. They should feel like a statement, not a SaaS template. Supporting copy should remain small and measured, usually 15–17px with generous line height.

Inside the app, use smaller type with clear weight changes rather than large sizes. Most operational text should live between 12px and 15px. Use muted foreground colors for labels, metadata, timestamps, breadcrumbs, and inactive controls.

## Layout & Spacing

The landing page uses a centered max-width layout with large vertical breathing room. The hero copy is centered in a wide text column, followed by CTAs, the works-with row, and one large product preview. Preserve this rhythm. Do not replace it with a split hero, multi-card cockpit, or crowded metric dashboard.

Body sections alternate between dark and light canvases. Section headings use the same serif display voice as the hero, while feature content uses clean grid cards with subtle borders. Dividers are usually created with one-pixel border grids and tonal background changes rather than heavy rules.

The application UI is compact and panel-based. Sidebars and property panels use fixed widths, hairline separators, and dense rows. Cards are rounded but not pillowy. The interface should feel efficient, not spacious for its own sake.

## Surfaces, Borders, and Depth

Dispatch relies more on **hairline borders and tonal separation** than on dramatic shadows. Landing buttons can use backdrop blur, but most components should remain solid and readable. Product preview screenshots can carry stronger elevation because they are hero objects.

Use borders at 8–14% opacity on landing surfaces and standard light borders in the app. Shadows should be soft and utilitarian. Avoid saturated colored glows around cards unless indicating a specific live agent state.

## Shape Language

The base radius is 10px. Buttons and cards generally sit between 10px and 16px. Badges and avatars are fully rounded. The product screenshot frame is intentionally sharp or minimally rounded so it feels like a captured application surface rather than a decorative card.

The Dispatch mark is an eight-point routing asterisk. It should be simple, monochrome, and current-color by default. The logo can spin during onboarding or first-run moments, but the landing header should usually keep it still.

## Motion

Motion is subtle and functional:

- Use 150–200ms transitions for hover, color, and background shifts.
- Use a 400ms fade-and-rise for onboarding or stepped flows.
- Use a 500ms spring pop only for success moments.
- Use ongoing pulse/impulse animation only when communicating active work.

Avoid decorative motion on the landing page unless it reinforces agent progress or system liveness.

## Component Guidance

### Landing Header

The header should feel lightweight and almost invisible over the hero. Use transparent background on dark hero routes, a 76px height, small wordmark, and compact buttons. The wordmark is lowercase “dispatch” with modest letter spacing.

### Hero Buttons

The primary hero button is white with ink text. The secondary button is a subtle ghost button with a dark translucent fill, white text, thin white border, and slight blur. This is intentionally restrained; do not change the primary CTA to a loud cyan button unless the entire landing palette is being redesigned.

### Product Preview

The product preview should show a real, readable Dispatch board or issue-management surface. It should use light UI, subtle column backgrounds, readable cards, assignees, statuses, priorities, and agent comments. It should not become a futuristic cockpit.

### Application Cards

Cards use white or semantic card surfaces, subtle borders, 14px radius, 16px padding, and compact type. Cards should feel like functional containers, not marketing tiles.

### Agent Activity

Agent activity should be visible through familiar product primitives: avatars, comments, status changes, tool-call rows, logs, and progress indicators. Use the bot icon and info color as accents. Keep the agent experience grounded in normal task management patterns.

## Implementation Notes for Future Screens

When creating new Dispatch screens or marketing assets, start with these questions:

1. Does this still feel like a calm developer productivity tool?
2. Is the hierarchy readable at a glance?
3. Are agents represented as teammates inside ordinary workflow surfaces?
4. Is the visual polish coming from spacing, typography, and contrast rather than effects?
5. Would this be believable as an open-source product a serious engineering team could self-host?

If the answer to any of those is no, simplify the design before adding more visual treatment.
